# 2026-07-23 - 编码器重构与二进制抽象

> 编码器不再摸裸指针——MessagePack 作为整体进出。

## 概览

编码器数据流重排、缓冲层抽象从 `std::string` 切为 `std::vector<std::byte>`。Decoder 骨架到位。

## 编码器数据流向重排

此前 `Encoder` 直接操作输出缓冲区裸指针：调用方分配一块连续内存，编码器在内部按偏移量分别写入帧头与帧体，写完后由调用方取走整块缓冲。这导致三个问题：

- **职责泄漏**：缓冲区生命周期由调用方管理，但缓冲区内部布局（帧头占前 N 字节、帧体紧随其后）的约定完全由编码器隐式支配，调用方必须"知道"这个布局才能正确分配大小；
- **写入顺序耦合**：帧体先于帧头写入（因为 `body_len_` 需在序列化后才能确定），但二者位于同一块缓冲的不同偏移——任一偏移算错即静默错位；
- **扩展困难**：将来若帧头尺寸变化（如引入扩展字段）、或帧体需要分段写入，调用方也必须同步修改缓冲分配逻辑。

### 现行流向

```
RequestMessagePack → Encoder::encode() → std::vector<std::byte> → Connections::send()
```

`encode()` 接收完整的 `RequestMessagePack` 引用与外部 `out_buffer`：

1. `body_encoder` 将 `RequestMsg` 经 nlohmann::json 序列化，写入 `out_buffer` 的 `no_padding_size` 偏移处，返回实际体长；
2. 将体长回填 `fh_.body_len_`；
3. `header_encoder` 将 opcode（1 字节）、status（1 字节）、`htonl(body_len_)`（4 字节）写入 `out_buffer` 前 6 字节。

帧头与帧体位于同一 `out_buffer` 内，但写入顺序为"先体后头"，由偏移量解耦——帧头固定 0 号位、帧体固定 `no_padding_size` 号位。调用方只需按 `no_padding_size + max_message_body_length` 分配一次，不再需要理解内部布局。

`encode()` 返回 `IoStatus`，`error` 表示 `MessagePack` 无效（opcode / status 未初始化、或 body 为空）。

#### 取舍

当前 `out_buffer` 仍由调用方在栈上分配（`core.cpp:169-170`），编码器不持有内部缓冲。这与注释中"在编码器缓冲区拼接，拼接完成之后写入输出缓冲区"的两段式设计尚有差距——目前是一段式直接写入调用方提供的缓冲。两段式的好处是编码器可以复用预分配的内部缓冲、省去每次 `encode()` 的分配开销，留待后续性能轮次落地。

## 二进制抽象的重新考量

此前读写缓冲及编码结果使用 `std::string` 承载。字符串在 C++ 中的语义是**文本**——它以 `char` 为元素类型、以 NUL 为合法终止符、受 SSO（Small String Optimization）影响布局，且标准明确允许 `c_str()` 与 `data()` 在 C++11 前不保证连续。

编码后的帧是**二进制字节流**：帧头包含 `htonl` 转换后的 4 字节大端长度，帧体是 UTF-8 JSON 的字节表示。字节只有在业务层（经 `nlohmann::json::parse` 还原为 `RequestMsg` 各字段）后才具备语义。用 `std::string` 承载这一中间表示存在三重问题：

- **语义错位**：读到 `std::string` 的调用方有理由假定其内容为文本——例如直接 `std::cout << buf`，但帧头的 4 字节 `body_len_` 是不可打印的大端整数，打印即乱码；
- **SSO 隐忧**：短帧（如仅含 `{"content":""}` 的响应）可能落入 SSO 阈值，`data()` 指向栈而非堆，与 `memcpy`/`send` 等底层原语混用时虽然合法但语义模糊，且不同标准库的 SSO 实现差异使依赖此行为的代码不可移植；
- **NUL 歧义**：帧体 JSON 不含嵌入 NUL，但帧头的大端 `body_len_` 高位字节在消息体较短时完全可能为 `0x00`。`std::string` 方法族中 `size()` 与 `strlen()` 的不对称在二进制上下文中是潜伏的错误源。

### 现行方案

全部缓冲容器改为 `std::vector<std::byte>`：

- `Connections::read_buf_` / `write_buf_`（`connections.h:36-37`）：`std::vector<std::byte>`
- `Encoder::encode()` 输出参数（`encoder.h:61`）：`std::vector<std::byte>&`
- 编码器内部 `body_encoder` / `header_encoder` 均操作 `std::vector<std::byte>&`

`std::byte` 是 C++17 引入的**严格二进制**类型：仅支持位运算、不允许算术、不可隐式转换为整数。用它对缓冲建模意味着"此处是字节序列，不要把它当文本读"。唯一的边界转换点在 `Connections::send()` ——它内部用 `reinterpret_cast<const std::byte*>` 写入写缓冲（`connections.cpp:69-71`），这是 `std::byte` 设计的合法用法。

## Decoder 骨架

`src/core/decoder.h` 落位：`class Decoder`，含空的 `decode()` 返回 `RequestMessagePack`。当前为骨架——实现了从 `read_buf_` 中切出完整帧、反序列化帧头与帧体的完整逻辑尚未填入。该骨架的存在使"编码器→连接→解码器"的闭环在模块层面已完成拓扑，不阻塞上层集成。

## 下一步

1. **帧校验机制**：当前编码与解码均无完整性校验——帧头不含 checksum / magic number / 帧尾定界符，无法检测传输错误、粘包半包。TCP 的字节流语义下，一次 `read()` 可能收到半帧、也可能收到一帧半，纯依赖"帧头中的 `body_len_` 指示读多少字节"的前提是该字段本身未被损坏或错位。计划方案：帧头增加 1 字节 magic（如 `0xCE`），帧尾增加 2 字节 CRC16（覆盖帧头+帧体），使解码器能够在三个层面判定帧边界——magic 定位帧头、`body_len_` 定界帧尾、CRC 验证完整性。

2. **零拷贝编码路径**：当前编码器写入调用方提供的 `std::vector<std::byte>`，再经 `conn.send()` 拷贝进 `write_buf_`（一次额外的 `insert`，`connections.cpp:69`）。对于高吞吐场景，中间缓冲可取消——`Encoder` 直接写入 `Connections::write_buf_` 的尾部，省去一次全帧拷贝。这要求校验机制先行到位：因为写入后不再有"回看并修正"的机会，若编码中途失败（如 JSON 序列化异常），写缓冲中已残留的不完整帧必须由校验层在接收端识别并丢弃，而不能依赖发送端回滚。

3. **Decoder 实现**：补齐 `decode()`，从 `read_buf_` 中按状态机提取完整帧——先定位 magic → 读帧头 6 字节 → 读 `body_len_` 字节帧体 → 校验 CRC → 反序列化 JSON → 组装 `RequestMessagePack`。粘包/半包由解码器内置的分帧状态机消化，对上层透明。
